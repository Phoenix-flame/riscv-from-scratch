# Step 38 — Memory-isolated processes: per-task page tables and a `satp` switch

Two earlier steps each built half of what an operating system needs to run untrusting programs side by side. The scheduler (Step 25) gave us *time* sharing: a timer interrupt preempts whichever task is running, saves its registers, and resumes another, so two tasks appear to run at once. The MMU (Steps 22 and 24) gave us *space* translation: a user-mode task's addresses pass through an Sv32 page table before reaching memory, so what the task calls address `X` need not be physical address `X`. This step puts the two halves together. Each task gets its own page table, and the context switch reloads `satp` — the register that points at the active page table — every time it changes who is running. The result is the thing the two halves were always pointing at: processes that share the CPU in time but are isolated in memory, unable to read or corrupt each other even when they use the very same addresses.

The striking part is that no hardware changed. The synthesizable MMU core already implements translation, machine and user privilege, the timer interrupt, and the `satp` CSR; the SoC already wires up a timer and a UART. Everything needed was present. The merge lives entirely in the kernel, which is the right place for it: isolation is a policy the software chooses by handing each task a different page table, and the hardware's only job is to enforce whatever table `satp` currently names.

## One address space per task

The demonstration runs two user tasks that are, by construction, nearly identical. Each owns a private data page that it maps at virtual address `0x1000`, and each has a stack mapped at virtual address `0x2000`. Those virtual addresses are the same in both tasks — and that sameness is the whole point. Task A's page table maps `0x1000` to one physical page; task B's maps the identical virtual address to a different physical page. When task A stores a value tagged with an `A` to `0x1000` and task B stores a value tagged with a `B` to *its* `0x1000`, the two stores land in different physical memory. Neither task can name, let alone touch, the other's page: there is no virtual address in task A's table that resolves to task B's physical page. Isolation here is not a convention the tasks agree to honor; it is the absence of any translation that would let one reach the other.

Building those tables is a few writes each. A task's root (level-1) table needs only three live entries: a pointer down to a level-0 table that carries the private page and the stack page as ordinary 4 KiB leaves, and a single 4 MiB superpage that maps the UART so the task can print. Every other entry stays zero, which in Sv32 means *invalid* — any access outside the handful of mapped pages faults. Because this SoC loads the program image into instruction ROM only, RAM comes up zeroed, so the kernel writes exactly the entries it wants and the rest are already invalid. The two tasks' tables sit at fixed physical addresses well above the kernel's own code, data, and stack, as do the physical pages they back.

A detail worth being honest about: only *data* addresses are translated on this core; instruction fetch uses physical addresses directly. So the two tasks share their code (it is the same physical instructions, fetched untranslated) and are isolated in their data and stacks. That is enough to demonstrate real memory isolation — a task cannot read or write another's data — but it is a simplification relative to a full system, where code would be translated too and each process could have a private code image. The boundary this step enforces is on data, which is where the interesting corruption would happen.

## The switch that makes it real

The scheduler's context switch already saved and restored thirty-two registers across a timer tick. Isolation adds exactly one more piece of saved state per task: its `satp` value, the Sv32-mode bit plus the physical page number of its root table. We store it in each task's context block alongside the saved registers, and the assembly switch, having chosen the next task to run, loads that task's `satp` and writes it before returning. From the instant of that write, every user-mode data access walks the new task's page table. Because this core keeps no TLB — it walks the page table on every translated access — the switch takes effect immediately; there is no cached translation to invalidate, so no `sfence.vma` is needed. (A core with a TLB would have to flush or tag it here; that is the cost the no-TLB design trades away.)

The ordering is quietly important. The switch runs in machine mode, where translation is off, so all of its own accesses — saving registers into the context block, reading the next `satp`, reprogramming the timer — are physical and unaffected by whatever value `satp` holds. Writing the new `satp` partway through the handler changes nothing about the handler's own behavior; it only arms the address space that the final `mret` drops into. So the kernel manipulates page tables and `satp` freely in machine mode and the new mapping comes alive precisely when control returns to the user task, not a moment before.

The handler also learned to tell a timer tick from a fault. The scheduler's original trap entry assumed every trap was the timer; now it checks `mcause`, and a trap that is not an interrupt — a task touching an unmapped address, say — is routed aside, recorded, and halted rather than mistaken for a scheduling event. In the isolation demo that path never fires, which the test confirms; its presence is what makes the isolation *enforced* rather than merely hoped for, since a stray access lands there instead of silently succeeding.

## What the demo shows

`make sched-mmu` runs the two tasks under the timer for a couple dozen ticks. The UART output interleaves `A` and `b`, the proof inherited from the scheduler step that the two tasks are genuinely preempting each other rather than one running to completion. Then the testbench reaches into physical memory and reads the two private pages directly. Task A's physical page holds a value tagged `A`; task B's holds one tagged `B`; the two are different. Had `satp` not been switched — had both tasks run under one page table — only a single physical page would ever have been written, and the other would have stayed zero. That task B's page is non-zero and carries B's own tag is the direct evidence that B ran under its own table: the switch happened, and the isolation held.

## What's verified here

The test checks both halves of the claim. Preemption: it counts the `A` and `b` characters the two tasks emit and confirms both ran and interleaved. Isolation: it reads the two tasks' private physical pages and confirms each contains its own task's tagged value, that the two pages differ, and that no page fault was recorded — i.e. both tasks stayed within their mappings, the same virtual address resolved to two distinct physical pages, and `satp` was genuinely reloaded across switches. The whole thing runs on the unmodified synthesizable MMU SoC; the step adds only the kernel, the context-switch change, and the testbench, so every prior test is undisturbed.

## Honest status

- Only data accesses are translated; instruction fetch is physical, so the two tasks share code and are isolated in data and stack. A full system would translate fetches too and give each process a private code image.
- There is no TLB — translation walks the page table on every access — which is why the `satp` switch needs no `sfence.vma`. A TLB would change both the performance story and the switch (it would require invalidation), and is the natural next step.
- Tasks run in user mode under a machine-mode kernel; there is no supervisor mode, so this is the M/U two-level model, not the M/S/U arrangement a Unix-like kernel uses. `satp` is managed from machine mode here.
- The scheduler is fixed at two tasks with static, hand-placed page tables and physical pages; there is no dynamic process creation, no demand paging, no copy-on-write, and the page faults that do occur are reported and halted rather than serviced. Those are the mechanisms a real virtual-memory OS adds on top of exactly this foundation.
- Verified in simulation only.

## Takeaway

Memory-isolated processes sound like a major new capability, and they are — but the mechanism is almost nothing once the pieces exist. Time sharing was already there in the scheduler; address translation was already there in the MMU; the only thing missing was the recognition that the active page table is *part of a task's context*, to be saved and restored like any register. Add `satp` to the context block, reload it in the switch, and two tasks that name the same addresses quietly stop being able to touch each other. The hardware enforces it; the kernel merely decides, one `satp` write at a time, whose world each task gets to see.
