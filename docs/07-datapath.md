# Step 07 — The single-cycle datapath (a working CPU)

This is where it all comes together. We instantiate the six tested blocks,
add the program counter and the "glue" logic that connects them — next-PC
selection, the branch comparator, the ALU operand muxes, and the write-back
mux — and produce a complete `cpu` module. Then we run a real program through
it and watch the registers come out right.

Files:
- `rtl/cpu.v` — the top-level CPU
- `sw/test_datapath.hex` — the test program (generated by a small encoder)
- `tb/cpu_tb.v` — runs the program and checks results
- `Makefile` — convenience build targets (new this step)

## The mental model: one instruction, one cycle

In a single-cycle design, the entire journey of an instruction happens in one
clock period as a wave of combinational logic, and the clock edge at the end
commits the results. The journey:

1. **Fetch** — read the instruction at `PC` from instruction memory.
2. **Decode** — split out the fields; the control unit and immgen react.
3. **Read registers** — `rs1`, `rs2` come out of the register file.
4. **Execute** — the ALU computes (arithmetic, or an address, or a compare).
5. **Memory** — loads read / stores write data memory.
6. **Write back** — the right value goes to `rd`.
7. **Next PC** — pick `PC+4`, a branch target, or a jump target.

Steps 1–6 are pure combinational logic. At the rising clock edge, three pieces
of state update *simultaneously*: the PC, the register file (if writing), and
data memory (if storing). That single edge is the heartbeat of the machine.

## The program counter

The PC is the only new state element here. It's a 32-bit register that resets
to a known address and otherwise loads `next_pc` each cycle:

```verilog
reg [31:0] pc;
wire [31:0] pc_plus4 = pc + 32'd4;

always @(posedge clk) begin
    if (rst) pc <= RESET_PC;
    else     pc <= next_pc;
end
```

## Wiring the blocks

Instantiating a sub-module and connecting it by name is the bulk of `cpu.v`.
For example, the register file:

```verilog
regfile u_regfile (
    .clk(clk), .we(reg_write),
    .rs1_addr(rs1_addr), .rs2_addr(rs2_addr),
    .rd_addr(rd_addr),   .rd_data(wb_data),
    .rs1_data(rs1_data), .rs2_data(rs2_data)
);
```

`regfile` is the module name, `u_regfile` is this instance's name, and each
`.port(signal)` ties a port to a wire in the datapath. The instruction-memory,
control, immgen, ALU, and dmem instances follow the same pattern. Field
extraction is just slicing the instruction:

```verilog
wire [6:0] opcode  = instr[6:0];
wire [4:0] rd_addr = instr[11:7];
wire [2:0] funct3  = instr[14:12];
wire [4:0] rs1_addr= instr[19:15];
wire [4:0] rs2_addr= instr[24:20];
wire       funct7b5= instr[30];
```

## The four muxes that make it programmable

A datapath is mostly fixed wiring plus a few **multiplexers** whose select
lines come from the control unit. Those muxes are what let one piece of
hardware behave differently per instruction.

**ALU operand A** — `rs1`, or the PC (for `auipc`):
```verilog
wire [31:0] alu_a = alu_src_a ? pc : rs1_data;
```

**ALU operand B** — `rs2`, or the immediate:
```verilog
wire [31:0] alu_b = alu_src_b ? imm : rs2_data;
```

**Write-back** — what value lands in `rd`:
```verilog
case (wb_sel)
    WB_ALU : wb_data = alu_result;  // arithmetic/logic
    WB_MEM : wb_data = mem_rdata;   // loads
    WB_PC4 : wb_data = pc_plus4;    // jal/jalr return address
    WB_IMM : wb_data = imm;         // lui
endcase
```

**Next PC** — covered below.

## The branch comparator

The control unit only flagged `branch`; the actual decision depends on
`funct3`. A small combinational comparator produces `branch_cond`:

```verilog
always @(*) case (funct3)
    3'b000 : branch_cond = (rs1_data == rs2_data);                  // beq
    3'b001 : branch_cond = (rs1_data != rs2_data);                  // bne
    3'b100 : branch_cond = ($signed(rs1_data) <  $signed(rs2_data)); // blt
    3'b101 : branch_cond = ($signed(rs1_data) >= $signed(rs2_data)); // bge
    3'b110 : branch_cond = (rs1_data <  rs2_data);                  // bltu
    3'b111 : branch_cond = (rs1_data >= rs2_data);                  // bgeu
    default: branch_cond = 1'b0;
endcase
wire branch_taken = branch & branch_cond;
```

`branch_taken` is true only when this *is* a branch **and** the condition
holds. Note the signed/unsigned split again — `blt` vs `bltu` is exactly why
we used `$signed` casts.

## Next-PC logic

Four possible next addresses, in priority order:

```verilog
wire [31:0] pc_target   = pc + imm;                  // jal & branches (PC-relative)
wire [31:0] jalr_target = {alu_result[31:1], 1'b0};  // (rs1+imm), LSB cleared

always @(*) begin
    if      (jalr)         next_pc = jalr_target;
    else if (jump)         next_pc = pc_target;      // jal
    else if (branch_taken) next_pc = pc_target;
    else                   next_pc = pc_plus4;
end
```

Two nice reuse tricks:
- **`jal` and branches share one adder** (`pc + imm`), since both are
  PC-relative — `imm` just carries the right (J or B) layout for each.
- **`jalr` reuses the ALU**: for `jalr` the control unit already set the ALU to
  compute `rs1 + imm`, so `alu_result` *is* the target; we only clear bit 0
  (the spec mandates the LSB be zeroed). No extra adder needed.

That's the whole CPU. No part of it is mysterious — it's the six blocks plus
these muxes and adders.

## The test program

`sw/test_datapath.hex` is generated by a small Python encoder so the machine
code is exact. In assembly it reads:

```asm
        addi x1, x0, 5      # x1 = 5
        addi x2, x0, 7      # x2 = 7
        add  x3, x1, x2     # x3 = 12
        sub  x4, x2, x1     # x4 = 2
        sw   x3, 0(x0)      # mem[0] = 12
        lw   x5, 0(x0)      # x5 = 12   (round-trips through data memory)
        beq  x1, x2, +8     # 5 == 7 ? no  -> falls through
        addi x6, x0, 99     # x6 = 99   (runs, because branch not taken)
        bne  x1, x2, +8     # 5 != 7 ? yes -> skip the next instruction
        addi x7, x0, 42     # x7 = 42   (SKIPPED by the taken branch)
        addi x8, x0, 1      # x8 = 1    (branch lands here)
        jal  x9, 0          # loop to self; x9 = return address (PC+4)
```

It deliberately exercises register arithmetic, immediates, a store/load round
trip, a not-taken branch, a taken branch (which must skip an instruction), and
a jump that captures a return address.

## Build and run

With the new Makefile:

```bash
make cpu
```

or by hand:

```bash
iverilog -g2012 -Wall -o build/cpu_tb.vvp \
    rtl/alu.v rtl/regfile.v rtl/imem.v rtl/dmem.v \
    rtl/immgen.v rtl/control.v rtl/cpu.v tb/cpu_tb.v
vvp build/cpu_tb.vvp
```

The PC trace shows the control flow happening for real:

```
  cycle 5: pc=00000018 instr=00208463   <- beq  (not taken)
  cycle 6: pc=0000001c instr=06300313   <- fell through to addi x6
  cycle 7: pc=00000020 instr=00209463   <- bne  (taken)
  cycle 8: pc=00000028 instr=00100413   <- jumped past x7's addi (0x24 skipped!)
  cycle 9: pc=0000002c instr=000004ef   <- jal to self
  cycle 10: pc=0000002c ...              <- looping
```

Notice `0x24` never appears — the taken `bne` skipped it. Then the register
check:

```
ok   x1 = 00000005    ok   x2 = 00000007    ok   x3 = 0000000c
ok   x4 = 00000002    ok   x5 = 0000000c    ok   x6 = 00000063
ok   x7 = 00000000    ok   x8 = 00000001    ok   x9 = 00000030
ok   mem[0] = 0c (the stored 12)
ALL TESTS PASSED
```

`x5 = 12` proves the store-then-load round trip through data memory. `x7 = 0`
proves the taken branch really skipped its `addi`. `x9 = 0x30` is the return
address the `jal` saved. The CPU works.

## A note on hierarchical access in the testbench

The testbench peeks at internal state with dotted paths:

```verilog
dut.u_regfile.regs[5]    // register x5 inside the regfile instance
dut.u_dmem.mem[0]        // byte 0 of data memory
```

This reaches through the instance hierarchy (`dut` → `u_regfile` → its `regs`
array). It's a simulation-only convenience for checking and debugging — you
can't do this in synthesizable code, but it's invaluable in testbenches and in
GTKWave.

## Look at it in GTKWave

```bash
make wave-cpu        # or: gtkwave build/cpu_tb.vcd
```

Add `clk`, `rst`, `pc`, `instr`, and expand `dut` → `u_regfile` → `regs` to
watch registers fill in. Add `branch_taken` and `next_pc` to see the PC choose
its path at each branch/jump. Stepping through, you can literally watch the
program execute.
